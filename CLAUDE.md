# Project Rules

## General Principles

- Follow the existing project architecture and conventions.
- Follow existing code patterns unless there is a strong reason not to.
- Maintain consistency with the surrounding codebase.
- Prefer simple solutions over complex solutions.
- Prefer the smallest change that correctly solves the problem.
- Avoid unnecessary abstractions.
- Avoid unnecessary dependencies.
- Avoid code duplication where practical.
- Preserve existing functionality unless explicitly instructed otherwise.
- Do not refactor unrelated code.

## Development

- Write clean, readable, and maintainable code.
- Write code suitable for production use.
- Consider edge cases and failure scenarios.
- Consider long-term maintainability.
- Prioritize reliability over cleverness.
- Reuse existing functionality when appropriate.

## Performance

- Consider performance implications of implementation decisions.
- Avoid unnecessary computation, rendering, or network requests.
- Optimize only when there is a clear benefit.

## Accessibility

- Consider accessibility when building user-facing features.
- Use semantic HTML where appropriate.
- Support keyboard navigation where applicable.

## SEO

- Consider SEO when relevant to the project.
- Prefer semantic markup and proper document structure.

## Design

- Prioritize clarity over visual complexity.
- Prioritize usability over visual effects.
- Maintain visual consistency.
- Use spacing, typography, and hierarchy intentionally.
- Design decisions should support the user's goals and the business goals.

## Decision Making

When multiple valid solutions exist, prioritize:

1. Correctness
2. Simplicity
3. Maintainability
4. User Experience
5. Performance

## Workflow

Before making significant changes:

1. Understand the existing implementation.
2. Identify constraints and risks.
3. Create a plan.
4. Implement the solution.
5. Review the result.

## Communication

- Explain important decisions.
- Explain tradeoffs when relevant.
- Identify risks before major changes.
- Do not assume requirements that were not specified.
- Ask for clarification when uncertainty could affect implementation.

## Before Completing A Task

- Review for correctness.
- Review for maintainability.
- Review for responsiveness when relevant.
- Review for accessibility when relevant.
- Review for SEO implications when relevant.
- Review for performance implications when relevant.
- Check for linting and type errors when applicable.

Project Context

This project is a premium digital business card website.

The design mockup provided by the user is the design specification.

The goal is to closely match the visual quality, layout, spacing, typography, hierarchy, and overall aesthetic of the mockup.

The website should feel:

* Premium
* Established
* Professional
* Minimal
* Trustworthy

The website is intentionally simple.

Do not add content that was not explicitly provided.

Do not invent:

* Services
* Practice areas
* Marketing copy
* Testimonials
* Case studies
* Blog content
* Appointment booking
* Call-to-action sections
* Additional pages

Pages:

* Homepage
* Impressum
* Datenschutzerklärung

Homepage content should only contain information provided by the client.

Design Priorities:

1. Visual quality
2. Typography
3. Spacing
4. Layout
5. Responsiveness

Prefer a refined, high-end appearance over strict minimalism.

Avoid pages that feel empty, unfinished, or generated.

The homepage should feel like a modern premium website, not a scanned letterhead.
